## 常见题目

### 1. Redis 是什么？为什么快？

Redis 是一个基于内存的高性能 Key-Value 数据库，常用于缓存、分布式锁、计数器、排行榜、消息队列、限流等场景。

Redis 快的主要原因：

- 数据主要存储在内存中。
- 使用高效的数据结构。
- 单线程执行命令，避免线程切换和锁竞争。
- I/O 多路复用，可以高效处理大量连接。
- 命令处理逻辑简单，执行路径短。

需要注意的是，Redis 的单线程主要指命令执行线程。新版本 Redis 在网络 I/O、持久化、异步删除等方面也会使用多线程。

### 2. Redis 常见数据类型有哪些？

常见数据类型包括：

- String：字符串、数字、二进制数据。
- Hash：对象字段存储。
- List：列表、简单队列。
- Set：无序集合、去重。
- ZSet：有序集合、排行榜。
- Bitmap：位图统计。
- HyperLogLog：基数统计。
- Stream：消息流。
- Geo：地理位置。

面试中要能说出每种类型的典型使用场景。

### 3. String 类型常见使用场景有哪些？

String 是最基础的数据类型，常见场景包括：

- 缓存对象 JSON。
- 计数器，例如浏览量、点赞数。
- 分布式锁。
- 限流计数。
- session 或 token 存储。

常用命令包括 `get`、`set`、`incr`、`decr`、`setnx`、`setex`。

### 4. Hash 类型适合什么场景？

Hash 适合存储对象结构，例如用户信息、商品信息、购物车条目等。

示例：

```text
user:1001
  name -> Tom
  age -> 20
  city -> Shanghai
```

相比把整个对象序列化成 JSON 存入 String，Hash 可以单独读写某个字段，减少不必要的数据传输和序列化开销。

### 5. List、Set、ZSet 分别适合什么场景？

List 是有序列表，适合简单队列、时间线、最近消息列表。

Set 是无序不重复集合，适合去重、共同好友、标签集合、抽奖候选池。

ZSet 是有序集合，每个元素有一个 score，适合排行榜、热度榜、延迟队列、范围查询。

### 6. Redis 的缓存穿透是什么？怎么解决？

缓存穿透指查询一个缓存和数据库都不存在的数据，请求每次都会打到数据库。

解决方式：

- 缓存空值，给空结果设置较短过期时间。
- 使用布隆过滤器，提前拦截不存在的 key。
- 做参数校验，拦截明显非法请求。
- 对异常访问做限流。

缓存空值实现简单，但会占用缓存空间；布隆过滤器性能好，但有一定误判率。

### 7. Redis 的缓存击穿是什么？怎么解决？

缓存击穿指某个热点 key 过期瞬间，大量请求同时访问，导致请求集中打到数据库。

解决方式：

- 热点 key 不设置过期时间，通过后台异步刷新。
- 使用互斥锁，只让一个线程回源数据库并重建缓存。
- 提前续期，在 key 过期前主动刷新。
- 使用逻辑过期，先返回旧数据，再异步更新缓存。

热点数据通常更适合逻辑过期或后台刷新。

### 8. Redis 的缓存雪崩是什么？怎么解决？

缓存雪崩指大量 key 在同一时间过期，或者 Redis 故障，导致大量请求直接打到数据库。

解决方式：

- 给过期时间增加随机值，避免同时失效。
- 热点数据分批预热。
- 使用多级缓存。
- Redis 做高可用部署。
- 数据库侧限流、降级、熔断。
- 对回源逻辑加互斥控制。

缓存雪崩重点是避免集中失效，并准备好降级方案。

### 9. Redis 有哪些持久化方式？

Redis 主要有两种持久化方式：

- RDB：定时生成内存快照，保存某个时间点的数据。
- AOF：记录每次写命令，通过重放命令恢复数据。

RDB 恢复速度快、文件紧凑，但可能丢失最近一次快照后的数据。AOF 数据更完整，但文件更大，恢复速度通常比 RDB 慢。

生产中常见做法是 RDB 和 AOF 混合使用。

### 10. RDB 和 AOF 有什么区别？

RDB 是快照持久化，保存的是某一时刻的数据状态。它适合备份和快速恢复，但实时性较弱。

AOF 是日志持久化，保存的是写命令。它数据安全性更好，可以配置每秒刷盘或每次写入刷盘，但文件体积更大，需要 AOF rewrite 压缩。

简单对比：

- RDB：恢复快、文件小、可能丢数据。
- AOF：数据更安全、文件大、恢复相对慢。

### 11. AOF 的刷盘策略有哪些？

AOF 常见刷盘策略：

- always：每次写命令都刷盘，安全性最高，性能最低。
- everysec：每秒刷盘一次，性能和安全性折中，最多丢失约 1 秒数据。
- no：由操作系统决定何时刷盘，性能高，但数据安全性较弱。

生产中常用 `everysec`。

### 12. Redis 的过期删除策略有哪些？

Redis 删除过期 key 主要使用两种策略：

- 惰性删除：访问 key 时检查是否过期，过期则删除。
- 定期删除：后台定期抽样检查并删除过期 key。

只靠惰性删除可能导致大量过期 key 占用内存；只靠定期删除又会增加 CPU 压力。所以 Redis 结合两种方式。

### 13. Redis 的内存淘汰策略有哪些？

当内存达到 `maxmemory` 限制时，Redis 会根据淘汰策略删除 key。

常见策略：

- noeviction：不淘汰，写入直接报错。
- allkeys-lru：在所有 key 中淘汰最近最少使用的 key。
- volatile-lru：在设置过期时间的 key 中淘汰最近最少使用的 key。
- allkeys-random：在所有 key 中随机淘汰。
- volatile-random：在设置过期时间的 key 中随机淘汰。
- volatile-ttl：优先淘汰快过期的 key。
- allkeys-lfu：在所有 key 中淘汰最少使用的 key。
- volatile-lfu：在设置过期时间的 key 中淘汰最少使用的 key。

缓存场景常见选择是 `allkeys-lru` 或 `allkeys-lfu`。

### 14. Redis 如何实现分布式锁？

最基本做法是使用 `SET key value NX EX seconds`：

```text
SET lock:order 123 NX EX 30
```

含义是只有 key 不存在时才设置成功，并设置过期时间，避免客户端宕机导致锁无法释放。

释放锁时必须先判断 value 是否是自己的，再删除 key。这个判断和删除需要保证原子性，通常使用 Lua 脚本实现。

### 15. 为什么 Redis 分布式锁需要设置唯一 value？

唯一 value 用来标识锁的持有者，避免误删别人的锁。

例如客户端 A 获取锁后业务执行太久，锁过期了。客户端 B 又获取了同一把锁。如果 A 之后直接 `del key`，就会把 B 的锁删掉。

所以释放锁前要判断 value 是否等于自己持有的 token，是自己的锁才能删除。

### 16. Redis 分布式锁有哪些问题？

常见问题包括：

- 锁过期时间不好设置，太短可能业务未完成锁就过期。
- 业务执行时间过长时需要续期。
- 主从切换时可能出现锁数据未同步导致多个客户端持锁。
- 释放锁必须校验 value，避免误删。
- 锁粒度过大会降低并发，过小又增加复杂度。

如果对一致性要求很高，可以考虑 ZooKeeper 锁或专业分布式协调方案。

### 17. 什么是 Redisson？它解决了什么问题？

Redisson 是 Java 常用的 Redis 客户端，提供了很多分布式对象和工具，比如分布式锁、读写锁、信号量、限流器、延迟队列等。

在分布式锁场景中，Redisson 提供了看门狗机制，可以在业务未执行完时自动续期，降低锁提前过期的风险。

### 18. Redis 事务是什么？

Redis 事务通过 `MULTI`、`EXEC`、`DISCARD`、`WATCH` 实现。

基本流程：

1. `MULTI` 开启事务。
2. 多条命令进入队列。
3. `EXEC` 执行队列中的命令。

Redis 事务可以保证命令按顺序执行，但不支持传统数据库那种完整回滚。如果某条命令运行时报错，其他命令仍可能继续执行。

### 19. Redis 的 Lua 脚本有什么作用？

Lua 脚本可以把多条 Redis 命令组合成一个原子操作执行。

常见场景：

- 分布式锁释放。
- 限流。
- 秒杀扣库存。
- 复杂计数逻辑。

Redis 执行 Lua 脚本时不会被其他命令插入执行，因此可以保证脚本内部逻辑的原子性。

### 20. Redis 主从复制是什么？

主从复制指一个 master 节点负责写入，多个 slave 节点复制 master 数据并提供读能力。

作用：

- 提升读性能。
- 做数据冗余。
- 支持故障恢复。

需要注意的是，Redis 主从复制是异步的，主从之间可能存在短暂数据延迟。

### 21. Redis 哨兵模式是什么？

哨兵模式用于监控 Redis 主从集群，并在 master 故障时自动完成主从切换。

哨兵主要职责：

- 监控 master 和 slave 是否存活。
- 判断 master 是否客观下线。
- 选举新的 master。
- 通知客户端新的 master 地址。

哨兵解决的是高可用问题，但不解决数据分片问题。

### 22. Redis Cluster 是什么？

Redis Cluster 是 Redis 官方集群方案，用于实现数据分片和高可用。

它将整个 key 空间划分为 16384 个 hash slot，每个 master 负责一部分 slot。客户端根据 key 计算 slot，再访问对应节点。

Redis Cluster 同时支持主从复制，master 故障时可以由 slave 升级为新的 master。

### 23. Redis Cluster 为什么是 16384 个槽？

Redis Cluster 使用 16384 个 hash slot 来管理数据分片。slot 数量固定后，节点扩容、缩容时只需要迁移部分 slot，而不是重新计算所有 key 的位置。

16384 是工程上的折中：数量足够细，可以较均匀地分配数据；同时元数据规模不会太大，便于节点间传播集群状态。

### 24. Redis 如何保证缓存和数据库一致性？

常见策略是先更新数据库，再删除缓存。

原因是直接更新缓存容易出现并发覆盖问题，而删除缓存可以让后续请求重新从数据库加载最新数据。

为了进一步降低不一致风险，可以结合：

- 延迟双删。
- binlog 监听异步删缓存。
- MQ 重试补偿。
- 设置合理过期时间。
- 对核心数据增加版本号或更新时间校验。

缓存和数据库很难做到绝对强一致，通常追求最终一致。

### 25. 为什么一般是删除缓存，而不是更新缓存？

更新缓存会遇到几个问题：

- 并发更新时，旧数据可能覆盖新数据。
- 有些缓存值需要复杂计算，更新成本高。
- 更新后的数据可能很快不用，造成无效写入。

删除缓存更简单，让下一次读取时再加载数据库并重建缓存，通常更稳妥。

### 26. Redis 的大 key 有什么问题？怎么处理？

大 key 指 value 很大，或者集合元素非常多的 key。

问题包括：

- 网络传输耗时长。
- 阻塞 Redis 主线程。
- 删除时可能造成卡顿。
- 主从同步和迁移成本高。
- 内存分布不均。

处理方式：

- 拆分大 key。
- 控制集合元素数量。
- 使用 `unlink` 异步删除。
- 避免一次性读取全部数据，改用分批扫描。
- 定期扫描和治理大 key。

### 27. Redis 的热 key 有什么问题？怎么处理？

热 key 指访问量特别高的 key。

问题包括：

- 单个 Redis 节点压力过大。
- 网络和 CPU 成为瓶颈。
- 极端情况下影响整个实例稳定性。

处理方式：

- 本地缓存。
- 多级缓存。
- 热 key 副本拆分。
- 读请求打散。
- 提前识别并预热。
- 对热点接口限流。

### 28. Redis 如何做限流？

常见限流方式：

- 固定窗口：用 `incr` 加过期时间实现，简单但边界可能突刺。
- 滑动窗口：用 ZSet 记录请求时间，更平滑但成本更高。
- 令牌桶：按固定速率生成令牌，请求消耗令牌。
- 漏桶：请求按固定速率流出。

简单业务可以用 `incr + expire`，高精度限流可以用 Lua 脚本或 Redisson 的限流器。

### 29. Redis 的 Stream 可以用来做什么？

Stream 是 Redis 5.0 引入的消息流数据结构，支持消息追加、消费组、消息确认等能力。

它可以用于轻量级消息队列场景，例如异步任务、事件通知、简单日志流。

但如果是大规模消息系统，通常还是会选择 Kafka、RocketMQ、RabbitMQ 等专业 MQ。

### 30. Redis 生产环境要注意什么？

常见注意点：

- 设置 `maxmemory` 和合理淘汰策略。
- 重要数据开启持久化。
- 避免大 key 和热 key。
- 避免 `keys *`、大范围 `hgetall` 等阻塞命令。
- 使用 `scan` 替代 `keys`。
- 合理设置过期时间，避免缓存雪崩。
- 做主从、哨兵或 Cluster 高可用。
- 监控内存、连接数、QPS、慢查询、命中率、复制延迟。
- 分布式锁要设置唯一 value，并用 Lua 释放。
- 对核心链路准备降级和限流方案。
