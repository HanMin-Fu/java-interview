## 常见题目

### 1. ZooKeeper 是什么？主要用来解决什么问题？

ZooKeeper 是一个分布式协调服务，核心能力是为分布式系统提供一致性数据存储和事件通知机制。

它常用于：

- 服务注册与发现
- 配置中心
- 分布式锁
- 集群选主
- 分布式协调
- 元数据管理

面试中可以这样概括：ZooKeeper 本身不是业务数据库，而是用来管理分布式系统中的协调状态，比如谁是主节点、服务是否在线、配置有没有变化。

### 2. ZooKeeper 的数据模型是什么样的？

ZooKeeper 的数据模型类似文件系统，是一棵树形结构。每个节点叫做 znode，每个 znode 都可以存储少量数据，也可以有子节点。

常见路径示例：

```text
/services/order-service/instance-1
/config/db
/locks/order-lock
```

需要注意的是，ZooKeeper 适合存储小数据和元数据，不适合存储大文件或大量业务数据。

### 3. ZooKeeper 有哪些节点类型？

ZooKeeper 常见节点类型有四类：

- 持久节点：客户端断开后节点仍然存在。
- 临时节点：客户端会话断开后节点自动删除。
- 持久顺序节点：持久节点，并且节点名后自动追加递增序号。
- 临时顺序节点：临时节点，并且节点名后自动追加递增序号。

临时节点常用于服务注册和心跳感知；顺序节点常用于分布式锁、队列、选主等场景。

### 4. 临时节点和持久节点有什么区别？

持久节点创建后会一直存在，除非主动删除。临时节点和客户端 session 绑定，客户端 session 失效后会被 ZooKeeper 自动删除。

因此，临时节点非常适合表达“在线状态”。例如服务实例启动时创建临时节点，实例宕机或网络断开导致 session 超时后，节点会自动消失，注册中心就能感知实例下线。

### 5. ZooKeeper 的 Watch 机制是什么？

Watch 是 ZooKeeper 的事件监听机制。客户端可以监听某个节点或子节点列表的变化，当节点发生创建、删除、数据变更、子节点变化时，ZooKeeper 会通知客户端。

它有几个特点：

- Watch 是一次性的，触发后需要重新注册。
- Watch 通知只告诉你发生了变化，不一定携带完整新数据。
- Watch 是异步通知，客户端收到通知后通常需要主动重新读取数据。

常见使用场景包括配置变更通知、服务上下线通知、锁释放通知。

### 6. 为什么 ZooKeeper 的 Watch 是一次性的？

一次性 Watch 可以降低服务端维护监听状态的复杂度，避免大量长期监听给 ZooKeeper 集群带来过重负担。

客户端收到通知后重新注册 Watch，可以让客户端明确表达自己仍然关心这个节点的后续变化，也能减少无效监听。

### 7. ZooKeeper 如何实现分布式锁？

典型做法是使用临时顺序节点：

1. 多个客户端在同一个锁目录下创建临时顺序节点。
2. 每个客户端获取当前目录下所有子节点并排序。
3. 序号最小的客户端获得锁。
4. 没拿到锁的客户端只监听自己前一个节点。
5. 当前一个节点删除后，再判断自己是否成为最小节点。
6. 业务执行完成后删除自己的临时顺序节点释放锁。

这种方式的优点是公平、有序，并且客户端宕机后临时节点会自动删除，不容易造成死锁。

### 8. 分布式锁为什么监听前一个节点，而不是监听锁目录？

如果所有客户端都监听锁目录，一旦锁释放，所有客户端都会收到通知并同时抢锁，容易产生羊群效应。

监听前一个节点可以让每次锁释放时只唤醒一个后继客户端，减少无效通知和竞争压力。

### 9. ZooKeeper 如何做服务注册与发现？

服务提供者启动时，在指定服务目录下创建临时节点，节点内容通常保存 IP、端口、权重等信息。

服务消费者监听服务目录的子节点变化：

- 有新节点创建，说明有新实例上线。
- 节点删除，说明实例下线。
- 节点数据变更，说明实例元数据变化。

消费者收到 Watch 通知后重新拉取服务列表，并更新本地缓存。

### 10. ZooKeeper 如何做配置中心？

可以把配置存储在指定 znode 上，客户端读取配置并注册 Watch。当配置节点数据发生变化时，客户端收到通知后重新读取配置。

不过 ZooKeeper 更适合小规模、关键元数据的配置管理。如果配置内容很大、变更很频繁，通常会选择专门的配置中心。

### 11. ZooKeeper 集群中有哪些角色？

ZooKeeper 集群主要有三类角色：

- Leader：处理写请求，负责事务提议和提交。
- Follower：参与投票，处理读请求，并转发写请求给 Leader。
- Observer：不参与投票，只同步数据和处理读请求，用于扩展读能力。

Leader 是写请求的核心，Follower 保证集群一致性，Observer 主要用于提升读性能，但不增加写事务投票成本。

### 12. ZooKeeper 为什么通常部署奇数个节点？

ZooKeeper 依赖过半机制，只要超过半数节点可用，集群就能正常工作。

例如：

- 3 个节点最多允许 1 个节点故障。
- 4 个节点最多也只允许 1 个节点故障。
- 5 个节点最多允许 2 个节点故障。

所以 4 个节点相比 3 个节点没有提升容错能力，反而增加成本。常见部署规模是 3 台或 5 台。

### 13. 什么是 ZooKeeper 的过半机制？

过半机制指一个事务操作必须获得超过半数节点确认，才会被认为提交成功。

例如 5 节点集群中，至少 3 个节点确认才算成功。这样可以保证任意两个合法多数派之间至少有一个交集，从而避免多个 Leader 同时提交互相冲突的数据。

### 14. ZooKeeper 如何避免脑裂？

ZooKeeper 通过过半机制避免脑裂。只有能获得超过半数节点支持的一侧才能选出 Leader 并继续对外服务。

如果网络分区后某一侧节点数量不足半数，即使里面有旧 Leader，也不能继续提交事务，只能停止写服务。这样可以避免两个分区各自成为主节点并写入冲突数据。

### 15. 什么是 ZAB 协议？

ZAB 是 ZooKeeper 的原子广播协议，用来保证写请求在 ZooKeeper 集群中的一致性。

它主要包含两个阶段：

- 崩溃恢复：Leader 选举和数据同步，保证新 Leader 拥有最新已提交事务。
- 消息广播：Leader 接收写请求，将事务 proposal 广播给 Follower，过半确认后提交。

面试中可以简化为：ZAB 保证 ZooKeeper 集群中写操作按顺序、可靠地复制到多数节点。

### 16. ZooKeeper 的写请求流程是什么？

写请求大致流程如下：

1. 客户端发起写请求。
2. 如果请求到 Follower，Follower 会转发给 Leader。
3. Leader 生成事务 proposal。
4. Leader 将 proposal 广播给 Follower。
5. 超过半数节点 ack 后，Leader 提交事务。
6. Leader 通知 Follower 提交。
7. 返回客户端写入成功。

ZooKeeper 的写操作由 Leader 统一协调，因此能保证事务顺序一致。

### 17. ZooKeeper 的读请求是否经过 Leader？

普通读请求可以由任意节点直接处理，不一定经过 Leader，所以 ZooKeeper 的读性能通常比写性能好。

但这也意味着普通读可能读到稍旧的数据。如果业务要求读取最新数据，可以在读之前调用 `sync`，让当前节点先和 Leader 同步。

### 18. ZooKeeper 保证强一致性吗？

ZooKeeper 对写请求提供顺序一致性和最终一致的数据复制，但普通读不一定是线性一致读。

简单说：

- 写请求由 Leader 串行处理，顺序有保证。
- 读请求可以落到任意节点，可能读到旧值。
- 如果需要更强的读一致性，可以配合 `sync`。

### 19. ZooKeeper 和 Eureka、Nacos 做注册中心有什么区别？

ZooKeeper 更强调 CP，即一致性和分区容错。在网络异常时，它更倾向于保证数据一致，可能牺牲部分可用性。

Eureka 更偏 AP，优先保证可用性，即使短时间内服务列表不完全准确，也尽量让系统继续可用。

Nacos 同时支持多种模式，在服务发现和配置中心场景下更贴近微服务生态，功能也更完整。

面试中可以结合业务说：如果更重视一致性和协调能力，可以考虑 ZooKeeper；如果是微服务注册发现，现代项目中 Nacos、Eureka、Consul 也很常见。

### 20. ZooKeeper 适合做哪些事情，不适合做哪些事情？

适合：

- 分布式协调
- 注册中心
- 配置元数据管理
- 分布式锁
- 集群选主
- 小数据量状态管理

不适合：

- 存储大文件
- 存储大量业务数据
- 高频写入场景
- 复杂查询场景

ZooKeeper 的定位是协调服务，不是通用数据库。

### 21. ZooKeeper 的 session 和心跳机制是什么？

客户端连接 ZooKeeper 后会建立 session。客户端需要定期发送心跳维持 session，如果超过 session timeout 仍未恢复连接，ZooKeeper 会认为 session 失效，并删除该 session 创建的临时节点。

这也是 ZooKeeper 能用临时节点感知服务上下线的基础。

### 22. 客户端断开连接后，临时节点会立即删除吗？

不会。临时节点不是 TCP 连接一断开就马上删除，而是在 session 超时后才删除。

如果客户端只是短暂网络抖动，并且在 session timeout 之前重新连接成功，临时节点仍然存在。

### 23. ZooKeeper 的节点数据有什么限制？

ZooKeeper 单个 znode 默认最大数据大小通常是 1MB。实际使用中不建议接近这个上限。

ZooKeeper 适合保存少量关键元数据，例如服务地址、配置版本、锁节点等，不适合保存大对象。

### 24. ZooKeeper 的 ACL 是什么？

ACL 是 ZooKeeper 的权限控制机制，用来控制客户端对 znode 的访问权限。

常见权限包括：

- CREATE：创建子节点
- READ：读取节点数据和子节点
- WRITE：修改节点数据
- DELETE：删除子节点
- ADMIN：修改 ACL

生产环境中，如果 ZooKeeper 暴露给多个系统使用，通常需要配置 ACL，避免误删或越权访问。

### 25. 生产环境使用 ZooKeeper 要注意什么？

常见注意点：

- 集群节点数一般使用 3 或 5。
- 不要存储大数据。
- 避免高频写入。
- 合理设置 session timeout。
- 客户端 Watch 触发后要重新注册。
- 分布式锁要监听前一个节点，避免羊群效应。
- 生产环境开启 ACL。
- 监控连接数、延迟、磁盘、内存、事务日志和快照。
- 定期清理历史快照和事务日志，避免磁盘被打满。

### 26. ZooKeeper 和 Redis 都能做分布式锁，有什么区别？

ZooKeeper 分布式锁通常基于临时顺序节点，天然支持锁释放、顺序排队和故障删除，语义更偏强一致协调。

Redis 分布式锁通常基于 `SET NX EX`，性能更高，实现更轻量，但需要处理过期时间、锁续期、误删锁等问题。

简单对比：

- ZooKeeper 锁一致性更强，适合协调要求高的场景。
- Redis 锁性能更好，适合高并发、短时锁场景。

选择时要结合业务对一致性、性能、复杂度的要求。
